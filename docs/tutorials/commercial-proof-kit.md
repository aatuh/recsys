# Tutorial: commercial proof kit
In this tutorial you will produce the local proof bundle a buyer should expect from a credible RecSys evaluation.


## Who this is for

- Developers proving the full loop before a pilot
- Product or engineering leaders who need concrete evaluation artifacts
- Security/procurement reviewers who want a reproducible local evidence trail

## What you will get

- A served `POST /v1/recommend` response from a synthetic ecommerce dataset
- A published artifact manifest generated by `recsys-pipelines`
- Eval-compatible exposure/outcome fixtures validated by `recsys-eval`
- JSON and Markdown offline evaluation reports
- A short decision-note template for ship/hold/rollback review

The dataset is synthetic and non-PII. Source files live under `examples/data/ecommerce-mini/`.

## Prereqs

- Docker + Docker Compose (v2)
- `make`
- `curl`
- `python3`
- Go toolchain

Verify:

```bash
docker compose version
make --version
curl --version
python3 --version
go version
```

## Verify the proof kit without Docker

From repo root:

```bash
make proof-kit-test
```

Expected:

- ecommerce fixtures validate against `recsys-eval` schemas
- an offline report is written under `tmp/commercial-proof-kit/eval/`
- pipelines produce a manifest under `tmp/commercial-proof-kit/pipelines/registry/current/demo/home/`
- object-store artifacts are written under `tmp/commercial-proof-kit/pipelines/objectstore/`

## Run the full local demo

From repo root:

```bash
bash scripts/commercial_demo.sh
```

The script starts the local Compose stack, configures the API with a temporary env file under
`tmp/commercial-proof-kit/api.env`, runs pipelines against the ecommerce fixture, publishes the manifest to MinIO,
calls `/v1/recommend`, and writes evaluation reports.

Expected output files:

```text
tmp/commercial-proof-kit/recommendation-response.json
tmp/commercial-proof-kit/served-exposures.eval.jsonl
tmp/commercial-proof-kit/manifest.json
tmp/commercial-proof-kit/eval/offline-report.json
tmp/commercial-proof-kit/eval/offline-report.md
tmp/commercial-proof-kit/decision-note.md
```

## Compare to the buyer deliverables pack

After the script finishes, compare your local files with the completed synthetic examples in the buyer deliverables
pack:

- Evaluation deliverables pack: [Evaluation deliverables pack](../for-businesses/evaluation-deliverables-pack.md)

The checked-in examples use stable excerpts from the same `ecommerce-mini` fixture and replace nondeterministic fields,
such as run IDs and timestamps, with placeholders.

By default the script stops the local Compose stack when it finishes. To keep the stack running for manual inspection:

```bash
KEEP_STACK=1 bash scripts/commercial_demo.sh
```

## Inspect the recommendation response

```bash
python3 -m json.tool tmp/commercial-proof-kit/recommendation-response.json
```

Expected:

- `items[]` is non-empty
- `meta.tenant_id` is `demo`
- `meta.surface` is `home`

## Inspect the evaluation report

```bash
sed -n '1,160p' tmp/commercial-proof-kit/eval/offline-report.md
```

Expected:

- a summary section
- offline metrics such as `precision@3`, `recall@3`, and `ndcg@3`
- data quality fields derived from joinable exposure/outcome fixtures

## Use the decision note

Open:

```bash
sed -n '1,200p' tmp/commercial-proof-kit/decision-note.md
```

Fill in:

- decision: ship / hold / rollback
- reasoning linked to the served response, manifest, exposure log, and eval report
- follow-ups before replacing the synthetic fixture with one real pilot surface
